# Программа и методика испытаний подсистемы «Шлюз»

Система контроля удалённого доступа на основе анализа траектории изменения сигнала
BLE‑меток. Документ составлен в соответствии с ГОСТ 19.301‑79.

---

## 1. Объект испытаний

Объект испытаний — программная подсистема **«Шлюз»** мобильного приложения
(вкладка «Шлюз»), обеспечивающая приём BLE‑меток, их идентификацию по белому
списку, оценку близости по уровню сигнала (RSSI) и выдачу команды открытия
во внешнюю систему (HTTP / TCP / MQTT) либо на контроллер HM‑10.

Версия приложения: **1.5.0+6**.

## 2. Цель испытаний

Подтвердить, что подсистема «Шлюз»:
- корректно отличает зарегистрированные («свои») метки от посторонних;
- принимает решение об открытии только при достаточной близости метки;
- защищена от случайных (одиночных, шумовых) срабатываний;
- соблюдает интервал блокировки повторного открытия;
- доставляет команду открытия выбранным транспортом и фиксирует результат
  в журнале событий.

## 3. Проверяемые требования

| Обозн. | Требование |
|--------|-----------|
| Т1 | Открытие выполняется только для метки, присутствующей в белом списке |
| Т2 | Сверка возможна по любому из полей: UUID, MAC, Major, Minor (логика «ИЛИ») |
| Т3 | Метка с уровнем сигнала ниже порога RSSI отбраковывается |
| Т4 | До открытия требуется не менее `samplesRequired` подтверждений за 5 с |
| Т5 | Повторное открытие блокируется на `cooldownSeconds` секунд |
| Т6 | Каждое решение и результат отправки фиксируются в журнале событий |
| Т7 | Команда открытия доставляется по HTTP/TCP/MQTT; ошибка доставки логируется |

## 4. Средства испытаний

**Аппаратные:**
- мобильное устройство № 1 (Android, роль «метка»);
- мобильное устройство № 2 (Android, роль «шлюз»);
- ПЭВМ (Windows) — приёмник команд для проверки транспорта;
- *(при наличии)* модуль HM‑10 и исполнительный механизм шлагбаума.

**Программные:**
- приложение (вкладки «Генератор» и «Шлюз»);
- эталонный приёмник вебхука (скрипт `recv.py`, см. п. 6, ТК‑07);
- утилита `ncat` / брокер `mosquitto` — для проверки TCP/MQTT (опционально).

**Исходная настройка стенда:**
1. Устройство № 1 → «Генератор» → формат **iBeacon**, `Major = 1`, `Minor = 7`;
   UUID зафиксировать. Запуск вещания.
2. Устройство № 2 → «Шлюз» → в белый список внесена запись «Тест‑1» с `Major = 1`.
3. Контроль присутствия метки в эфире выполняется вкладкой «Сканер».

> Примечание. Управление дальностью срабатывания при стендовых проверках
> производится изменением порога RSSI; проверка реальной близости —
> физическим перемещением устройства‑метки.

## 5. Порядок проведения испытаний

Испытания проводятся последовательно по тест‑кейсам ТК‑01…ТК‑09 (п. 6).
Результат каждого тест‑кейса заносится в протокол (п. 7). Тест‑кейс считается
пройденным, если фактический результат совпадает с ожидаемым. При расхождении
фиксируется замечание; после устранения тест‑кейс повторяется.

## 6. Методы испытаний

**ТК‑01. Открытие для зарегистрированной метки (Т1, Т6).**
Исходные данные: порог RSSI = −90, метка рядом.
Порядок: запустить шлюз кнопкой «Старт».
Ожидаемый результат: в журнале появляются `Кандидат: Тест‑1 (1/2, RSSI=…)`,
затем `Открытие: Тест‑1 · Major · RSSI=…` (уровень «успех»).

**ТК‑02. Игнорирование посторонней метки (Т1).**
Исходные данные: вещание «своей» метки выключено либо используется
незарегистрированное устройство; в «Сканере» метка видна.
Порядок: наблюдать журнал шлюза 15 с.
Ожидаемый результат: строка `Открытие` отсутствует.

**ТК‑03. Контроль близости по RSSI (Т3).**
Исходные данные: порог RSSI = −55.
Порядок: удалить метку на расстояние (сигнал слабее порога), затем поднести вплотную.
Ожидаемый результат: на удалении — записей нет; при поднесении — `Открытие`.

**ТК‑04. Защита от одиночных срабатываний (Т4).**
Исходные данные: `samplesRequired = 5`, порог −90.
Порядок: запустить шлюз, метка рядом.
Ожидаемый результат: последовательность `Кандидат … (1/5 … 4/5)`,
команда `Открытие` формируется только при достижении 5/5.

**ТК‑05. Блокировка повторного открытия (Т5).**
Исходные данные: `cooldownSeconds = 15`.
Порядок: добиться открытия; не убирая метку, наблюдать журнал.
Ожидаемый результат: повторное `Открытие` не возникает ранее 15 с.

**ТК‑06. Сверка по отдельным полям (Т2).**
Исходные данные: в записи белого списка оставлено только `Minor = 7`
(поля UUID и Major очищены).
Порядок: запустить шлюз, метка рядом.
Ожидаемый результат: `Открытие: Тест‑1 · Minor · RSSI=…`.
Проверку повторить, оставляя поочерёдно только UUID, только Major, только MAC.

**ТК‑07. Доставка команды по HTTP (Т7, Т6).**
Исходные данные: транспорт = HTTP, адрес приёмника `http://<IP‑ПЭВМ>:8000`.
На ПЭВМ запущен приёмник:
```python
from http.server import BaseHTTPRequestHandler, HTTPServer
class H(BaseHTTPRequestHandler):
    def do_POST(self):
        n = int(self.headers.get('Content-Length', 0))
        print('ОТКРЫТИЕ →', self.rfile.read(n).decode())
        self.send_response(200); self.end_headers()
HTTPServer(('0.0.0.0', 8000), H).serve_forever()
```
Порядок: добиться открытия (ТК‑01).
Ожидаемый результат: в журнале шлюза — `HTTP: 200 OK`; на ПЭВМ в консоли —
JSON с полями `vehicle`, `rssi`, `timestamp`.

**ТК‑08. Обработка ошибки доставки (Т7).**
Исходные данные: транспорт = HTTP, заведомо недоступный адрес.
Порядок: добиться открытия.
Ожидаемый результат: запись уровня «ошибка» — `HTTP: таймаут (5 сек)`
либо код состояния ответа.

**ТК‑09. Сохранение конфигурации.**
Исходные данные: изменены порог RSSI, `cooldown`, `samplesRequired`, белый список.
Порядок: закрыть и повторно открыть приложение.
Ожидаемый результат: введённые параметры и список меток восстановлены.
